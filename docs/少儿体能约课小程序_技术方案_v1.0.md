## 1. 约束与决策依据

1. **硬件**：阿里云 2c4g / 40G 盘单机（116.62.34.224），数据库、后端、Nginx 全部同机部署，内存是最稀缺资源。
   - ⚠️ **当前系统为 CentOS 7，已于 2024-06-30 EOL**（无安全补丁、yum 源归档、新版 Docker 安装困难），必须在部署前重装。推荐 **Ubuntu 24.04 LTS**（社区资料最多，学习友好）或 Alibaba Cloud Linux 3（阿里云官方维护）；趁未承载生产负载，在阿里云控制台直接更换系统镜像即可。
2. **规模**（来自 PRD 10.2 性能假设）：单机构学员 ≤ 500、周课次 ≤ 100、并发峰值 ≤ 50 QPS——任何主流技术栈都能轻松扛住，**选型的决定因素不是性能，而是"一个人长期维护的成本"**。
3. **团队**：一人开发，前端背景，正在按 6 个月路线学习后端（Python 前 3 个月 → Go 后 3 个月）。
4. **产品性质**：给真实机构用的生产系统，有真实家长数据，要长期稳定运行，不是练完即弃的 demo。
5. **微信硬性要求**：小程序 request 合法域名必须是 HTTPS + ICP 备案域名——你的已备案域名满足条件。

## 2. 后端选型：为什么是 Go 而不是 Python

两者都能做好这个项目，差异在工程权衡：

| 维度 | Go（Gin + GORM） | Python（FastAPI + SQLAlchemy） | 谁占优 |
| --- | --- | --- | --- |
| 内存占用 | 单进程 ~50-80MB | uvicorn 多 worker ~300-500MB | Go，对 4G 单机很实在 |
| 部署形态 | 编译为单二进制，扔上去就跑 | 需要 Python 运行时 + 依赖管理 | Go，运维心智负担小 |
| 长期稳定性 | 静态类型，状态机/扣课规则这类业务在编译期挡住一批错误 | 动态类型 + Pydantic 可弥补，但运行期才暴露 | Go |
| 开发速度 | 样板代码多，起步慢 | 极快，自动 OpenAPI 文档 | Python |
| 上手曲线（前端背景） | 语法简单但理念（指针/接口/error）需适应 | 最平缓 | Python |
| 微信生态 SDK | silenceper/wechat 成熟可用 | wechatpy 成熟可用 | 平手 |
| 与你的学习终点 | 路线最终落点就是 Go | 路线起点 | Go |

**决策：Go。** 三个决定性理由：

1. **不重写**。这是要长期服务真实机构的系统。如果按学习路线先用 Python 写，3 个月后再"用 Go 重写"，等于对一个有真实数据、真实用户的生产系统做整体迁移——风险和浪费都不可接受。生产系统只选一次语言。
2. **一个人的运维**。Go 交付物是一个二进制 + 一个配置，没有 venv、依赖冲突、worker 调优；进程崩了 systemd/Docker 拉起来就好。对"开发者 = 运维 = 你自己"的局面，这是实打实的省心。
3. **机器小**。省下来的几百 MB 内存留给 PostgreSQL 的缓存，比留给解释器值钱。

**代价（诚实声明）**：如果你目前还在学习路线的 Python 阶段甚至更早，直接上 Go 意味着打乱"先 Python 后 Go"的顺序，且 PRD 写的"4-6 周 MVP"对边学边写的人**不现实，实际预期 10-14 周**。建议把本项目直接作为学习路线 Go 阶段的主线实战项目（替代路线里的"短链接服务"），路线中 Python 阶段可压缩或并行只学通用知识（HTTP、SQL、数据库设计——这些本项目全用得上）。

| 组件 | 选择 | 说明 |

### 2.2 为什么不用 Redis（MVP 阶段）

缓存解决的是"读压力大且业务容忍短暂不一致"的问题，本项目两个条件都不成立：

1. **没有读压力**：全部业务数据 < 50MB，PostgreSQL 的 shared_buffers 即可全库常驻内存，每次查询本身就是内存读；峰值 50 QPS 离 PG 在 2c4g 上的能力（数千 QPS）差两个数量级。
2. **核心数据不允许缓存**：课时余额、剩余名额是本产品的命脉（家长质疑余额是机构最大纠纷源），一旦进缓存就引入失效与不一致问题——强一致数据不缓存是原则，不是性能取舍。防超卖必须落在数据库原子操作上（见 6.1），"Redis 预扣库存"是万人秒杀的方案，8 人小班用它是引入复杂度而非解决问题。
3. **常见用途均有单实例替代**：微信 access_token → 进程内存变量；接口限流 → 进程内令牌桶；session → JWT 无状态。
4. **成本真实存在**：4G 内存上 Redis 占 50-100MB+，多一个容器、故障点、备份与监控对象，以及缓存一致性 bug 的排查成本。

**结论：MVP 引入 Redis 是过度工程**，触发条件见第 9 章（多实例 / 实测慢查询 / SaaS 阶段）。如以学习 Redis 为目的，建议在统计看板等非关键路径练手，不进入余额与名额链路。

### 2.3 为什么是 PostgreSQL 而不是 MySQL

两者都行，选 PostgreSQL 的理由：① **部分唯一索引**（partial unique index）一条索引就解决"同一学员同一课次只能有一条 booked 预约"的幂等问题（MySQL 要用状态冗余列绕）；② 你的学习路线教的就是 PostgreSQL，知识直接复用；③ 单机资源占用两者相当。

## 3. 前端选型

### 3.1 小程序（家长端 + 教练端）：Taro 4 + React

- **Taro 4 + React + TypeScript + NutUI React（Taro 适配版）组件库**（京东官方维护、Taro 官方文档推荐，含日历、单元格列表、徽标等本项目刚需组件）。
- 选型理由：开发者明确偏好 React 技术栈、不使用原生小程序写法；Taro 是 React 写小程序最成熟的方案（京东系大规模生产验证）。统一 React 后的附加收益：与管理后台共用语言心智、Hooks 模式、ESLint/TS 配置，request 封装与类型定义（API 接口 TS 类型）可在两端复用。
- 家长端与教练端**同一个小程序**，登录后按身份（手机号匹配教练档案）切换视图，符合 PRD FR-C-001。
- 状态管理：页面少、状态简单，用 React Context + Hooks 即可（当前孩子、登录态），不引入 Redux；如后续状态变复杂，加 Zustand（与后台一致）。
- 网络层：基于 `Taro.request` 封装统一请求（拼 token、统一错误码处理、401 跳登录），一个文件解决。
- **风险与纪律（跨端层的代价，需自觉遵守）**：
  - 锁定 Taro 小版本（pnpm lockfile + 固定版本号），升级单独开分支验证；
  - 排障时先分清问题在 Taro 编译层还是微信小程序层（用微信开发者工具看编译产物）；
  - 只投微信端，**不为"将来可能的多端"做任何额外抽象**——用 Taro 是为了 React 写法，不是为了跨端；
  - 涉及微信专属能力（订阅消息授权、手机号快速验证组件）直接用 Taro 暴露的微信 API，不找跨端等价封装。

### 3.2 管理后台：Ant Design Pro 模板

- **Ant Design Pro（umi 4 / @umijs/max + React + TypeScript + Ant Design 5 + ProComponents）**，官方脚手架 `create-umi` 初始化。
- 选型理由：开发者要求用成熟模板降低封装成本。Pro 模板开箱即用的部分恰好是后台的脏活：登录页与鉴权流程、整体布局与菜单、基于 access 的角色权限、请求封装与统一错误处理；**ProTable / ProForm / ProDescriptions 对本项目这种 CRUD 密集型后台（学员/教练/课程/课时卡/通知共约 10 个列表页）是最大提效点**——一个配置对象出一张带搜索、分页、操作列的完整表格。
- **配套约束（用模板就贯彻模板的体系，不混搭）**：
  - 请求层用模板内置的 umi request + ahooks useRequest，**不再引入 TanStack Query 和 Axios**；
  - 全局状态用 umi 的 useModel（initialState 存登录用户），**不再引入 Zustand**；
  - 权限用模板的 `access.ts`，按 admin/staff 两种角色写访问控制，对接后端 JWT 的 role。
- **初始化后第一件事是做减法**，删掉不用的部分：国际化（locales）、mock 目录（直连真实后端）、示例页面、OpenAPI 生成器之外的冗余配置——模板的成本主要来自没删干净的样板。
- 排课日历组件：仍 MVP 自绘周视图（7 列 × 时段行的 CSS Grid），不引入 FullCalendar，ProComponents 没有现成周视图排课组件，这一页是全后台唯一需要真正自己写的页面。
- 风险与纪律：umi 是黑盒度较高的框架，锁定 @umijs/max 小版本；遇到构建/路由疑难先查 umi 文档而非通用 React 方案（两者解法常不同）。
- 构建产物仍为纯静态文件（`max build`），由 Nginx 直接托管，不需要 Node 服务端。

## 5. 部署方案（2c4g 资源预算）

### 5.1 内存预算

| 进程 | 预算 | 说明 |
| --- | --- | --- |
| 系统 + Docker | ~500MB | |
| PostgreSQL | ~800MB | shared_buffers=256MB，max_connections=50 |
| Go API | ~100MB | |
| Nginx | ~30MB | |
| 余量 | ~2.5GB | 给 pg 页缓存和突发；另配 2GB swap 兜底 |

### 5.2 部署形态

- 一个 `docker-compose.yml` 管三个容器：`nginx`、`api`、`postgres`；pg 数据挂载到宿主机卷。
- Go 镜像用多阶段构建（builder + scratch/alpine），最终镜像 ~20MB。
- 管理后台构建产物放 Nginx 静态目录，不单独起容器。
- 部署流程（起步版）：本地 `make deploy` = 交叉编译/构建镜像 → scp 到服务器 → `docker compose up -d`。稳定后迁移到 GitHub Actions 自动化。

### 5.3 HTTPS 与域名

- acme.sh 申请 Let's Encrypt 通配符或两条单域名证书，cron 自动续期，Nginx 引用。
- 小程序后台配置 request 合法域名 `https://api.yourdomain.com`（要求备案 ✅ + HTTPS ✅ + 443 端口）。

### 5.4 安全基线

- 阿里云安全组只开 22（SSH）、80（仅跳转 443）、443；**PostgreSQL 端口不映射到宿主机公网**，仅 compose 内网访问。
- SSH 改密钥登录、禁密码；服务器上不放微信 AppSecret 之外的多余密钥，AppSecret 走环境变量不进 git。
- API 层：所有接口服务端鉴权（PRD 红线），登录接口限流（进程内令牌桶），bcrypt 哈希后台密码。
- 数据：每日 3:00 `pg_dump` 压缩后传阿里云 OSS，保留 30 天；每月手动做一次恢复演练（备份不演练等于没有）。

## 6. 关键技术设计

### 6.1 预约防超卖（PRD 5.1 / E-01）

名额扣减不依赖 ORM 读改写，用一条原子 UPDATE 在事务内完成：

```sql
BEGIN;
-- 1. 原子抢名额：影响行数为 0 即已满
UPDATE class_session
SET booked_count = booked_count + 1
WHERE id = $1 AND booked_count < capacity AND status = 'scheduled';
-- 2. 影响行数=1 才继续插入 booking，否则回滚返回"已约满"
INSERT INTO booking (...) VALUES (...);
COMMIT;
```

### 6.2 重复预约幂等（PRD E-02 / E-13）

PostgreSQL 部分唯一索引，数据库层兜底（应用层校验只是提前给出友好提示）：

```sql
CREATE UNIQUE INDEX uniq_active_booking
ON booking (class_session_id, student_id)
WHERE status = 'booked';
```

家长端连点由前端按钮禁用 + 该索引双重保证，无需引入分布式锁/请求去重表。

### 6.3 扣课一致性（PRD 5.3）

签到、撤销、调整全部走"同事务内：更新卡余额 + 写 credit_transaction 流水"，流水表带变动前后余额。任何课时变动不允许绕过该 service 函数——这是代码评审红线。

### 6.4 微信集成

- 登录：`wx.login` → 后端 code2session 换 openid → 签发 JWT。手机号用微信手机号快速验证组件获取（getPhoneNumber）。
- access_token：silenceper SDK 内置内存缓存，单实例无共享问题。
- 订阅消息：发送失败（用户未授权/配额耗尽）静默降级记日志，不影响主流程（PRD FR-P-013）。

### 6.5 鉴权与权限

- 单 user 表多角色（admin / staff / coach / parent），JWT claim 带 role + merchant_id。
- Gin 中间件链：`Auth（解析 JWT）→ RequireRole（角色门禁）→ handler 内数据范围校验`（家长只能查绑定学员、教练只能查自己课次——这层必须在 service 内做，不能只靠中间件）。

### 6.6 错误码与响应规范

```json
{ "code": 0, "message": "ok", "data": {} }
```

业务错误码分段：1xxx 通用、2xxx 预约（2001 已约满、2002 课时不足、2003 重复预约、2004 已截止…）、3xxx 课时卡、4xxx 权限。小程序端按 code 给 PRD 要求的"明确原因 + 下一步"提示。

```text
│   │   ├── api/              # 接口定义 + TS 类型（与 admin 共享类型定义）
│   │   └── utils/request.ts  # Taro.request 统一封装
│   └── config/               # Taro 编译配置（锁定版本）
├── admin/                    # Ant Design Pro 管理后台
│   └── src/
│       ├── pages/            # 按 PRD 13.3 页面清单，ProTable/ProForm 为主
│       ├── services/         # umi request 接口层（TS 类型与小程序共享）
│       ├── access.ts         # 角色权限（admin/staff）
│       └── app.tsx           # 运行时配置：请求拦截、initialState（登录用户）
├── deploy/
│   ├── docker-compose.yml
│   └── nginx.conf
└── docs/                     # PRD、本方案、API 文档
```

分层纪律（也是学习重点）：handler 只做参数绑定和响应，**所有业务规则在 service 层**（这样 PRD 第 12 章的 GWT 验收用例可以直接写成 service 层的表驱动测试，不经过 HTTP）。

## 8. 工程规范（与学习路线对齐）

- Git：main 保护，feature 分支 + 小步提交（路线要求每周 ≥5 次提交，本项目天然满足）。
- 测试底线：预约 service、扣课 service 的测试覆盖 PRD 第 12 章全部 12 条 GWT 用例 + 第 8 章 E-01~E-14 边界场景，其余模块尽力而为。
- `make lint`（gofmt + go vet）进提交习惯；CI 后补。
- 每个模块完成后用 curl/Apifox 全量过一遍接口（路线"不要跳过的训练"第 1 条）。

## 9. 演进路线（什么时候才需要加东西）

| 触发条件 | 动作 |
| --- | --- |
| 上线前 | AutoMigrate 切换为 golang-migrate 版本化迁移 |
| 多商户 SaaS（PRD V0.4） | 加 Redis（共享 access_token、缓存）；考虑 API 双实例 + 负载均衡 |
| 学员数 > 5000 或 QPS > 500 | 读写分离/换更大机器（优先加钱换 4c8g，比改架构便宜） |
| 需要图片上传（学员头像等） | 直接用阿里云 OSS + STS 直传，不过 API 中转 |
| 公众号模板消息（P2 触达备选） | 接入认证公众号，复用 silenceper SDK |

明确**不做**的清单：微服务、消息队列、Kubernetes、读写分离、分库分表——在可预见的业务量内全部是负资产。

## 10. 风险与对策

| 风险 | 对策 |
| --- | --- |
